# 🏗️ SSPA Technical Architecture Specification (v1.0.0)

Dokumen ini mendefinisikan arsitektur sistem tingkat tinggi, skema data kriptografis, model operasional node, serta protokol pemulihan untuk **SSPA Human Rights Ledger**. Desain ini dibangun menggunakan prinsip *privacy-by-design*, *offline-first survivability*, dan tata kelola berbasis aturan hukum (*rule-of-law-governed digital trust*).

---

## 1. Topologi Jaringan & Data Flow Diagram

SSPA beroperasi pada jaringan hibrida terdistribusi yang memisahkan data identitas sensitif penyintas dari catatan pembuktian yang tidak dapat diubah (*immutable ledger anchors*).

## 2. Spesifikasi Komponen Inti

### 2.1 Survivor Identity Vault (SIV)
*SIV* adalah wadah data terenkripsi *offline-first* berbasis klien (*client-side*) yang mengisolasi Pengenal Unik Dekentralisasi (DIDs), metadata kedaulatan, dan pointer asinkron ke berkas bukti.

*   **Prinsip Enkripsi**: Enkripsi simetris menggunakan `AES-256-GCM` dengan kunci enkripsi yang diturunkan dari frasa mnemonik lokal menggunakan `PBKDF2`.
*   **Aparatus Kriptografi**: Kunci publik penandatanganan menggunakan kurva eliptik `Ed25519` untuk mengesahkan integritas data yang dikirim tanpa membuka identitas asli klien.

### 2.2 Local Evidence Hash Store
Penyimpanan lokal di tingkat *edge device* (ponsel lapangan agen atau aplikasi penyintas) bertindak sebagai log transaksi berbasis *append-only*. 
*   Setiap data atau berkas digital dikonversi secara instan menjadi string hash kriptografis `SHA-256`.
*   Aplikasi menerapkan tanda tangan digital berbasis stempel waktu (*tamper-evident cryptographic timestamp*) menggunakan *state-replication* melalui model data CRDT (*Conflict-free Replicated Data Type*) untuk memastikan konsistensi data log lokal tanpa benturan data.

### 2.3 Distributed Ledger Anchors
Lapisan penambatan (*anchor layer*) SSPA menggunakan rantai terdistribusi ringan yang kompatibel dengan arsitektur EVM (Ethereum Virtual Machine).
*   **Kapasitas Minimal Data**: Tidak ada Informasi Identitas Pribadi (PII) yang diunggah ke *on-chain*. Ledger hanya mencatat root pohon Merkle (*Merkle Root*) dari kumpulan bukti kemanusiaan serta referensi ID validator yang memprosesnya.
*   **Interoperabilitas**: Smart contract SSPA dirancang berbasis standar *ERC-20 Compliant* untuk memfasilitasi pembatasan alokasi genesis, *vesting schedule*, dan pendanaan *Community Treasury* menggunakan mekanisme ekosistem kustom berbasis fungsionalitas non-spekulatif.

## 3. Matriks Peran Node & Operational Playbook

| Tipe Node | Entitas Operator | Peran Sistem | Persyaratan Bandwidth & Hardware |
| :--- | :--- | :--- | :--- |
| **Mobile Edge Node** | Penyintas / Agen Lapangan | Ingestion, Client Hashing, SIV Local Storage | **0 Kbps (Offline-Native)**. RAM minimal 2GB, Android 8+ / iOS 13+. |
| **Local Relay Hub** | Kantor Lapangan NGO / Pos Darurat | Store-and-Forward, Ad-hoc Mesh Routing, Local Aggregation | **Intermittent (128 Kbps - 1 Mbps)**. Wi-Fi Direct, Bluetooth 5.0, Raspberry Pi 4 / Laptop Medis. |
| **Validator Node** | Lembaga Kemanusiaan Internasional / Universitas | Konsensus, State Verification, Multi-Sig Treasury Management | **Continuous (10 Mbps+)**. Server dedicated 4 Cores, 16GB RAM, 1TB SSD, Akses internet stabil. |

## 4. Protokol Pemulihan & Penambatan Data Langkah-Demi-Langkah

### 4.1 Skema Skalabilitas Skema API / Format Anchor
Setiap transaksi penambatan ke jaringan inti SSPA wajib mematuhi payload skema biner terstruktur berikut:

```json
{
  "\$schema": "https://quorumstate.international",
  "chainId": 8080,
  "anchorMetadata": {
    "merkleRoot": "0x8f3a9c7d4b2e1f6d9a0c5b7e2d4f8a9b3c7d1e5f6a8b9c0d1e2f3a4b5c6d7e8f",
    "batchSize": 150,
    "blockTimestamp": 1774056000,
    "validatorSignature": "0x9c8b7a...2f1e"
  },
  "governanceControls": {
    "treasuryAllocationWei": "0",
    "activeVestingBlock": 2450000
  }
}
```

### 4.2 Offline Rebind & Social Recovery Protocol
Ketika perangkat keras penyintas hancur atau disita, proses penautan ulang identitas (*identity rebind*) berjalan melalui mekanisme *Guardian Pattern* yang ketat:

1.  **Inisiasi**: Penyintas mendatangi *Local Relay Hub* terafiliasi dan memasukkan frasa mnemonik parsial yang mereka hafal.
2.  **Penerbitan Tantangan ZKP**: Sistem memicu fungsi *Zero-Knowledge Proof* (ZKP) untuk membuktikan kepemilikan hash SIV lama tanpa memerlukan pengungkapan data demografis.
3.  **Atestasi Sosial (*Guardian Attestation*)**: Minimum 3 dari 5 institusi kemanusiaan (Validator Nodes) yang bertindak sebagai pemegang potongan kunci (*shard anchors*) wajib menandatangani transaksi persetujuan menggunakan skema *Threshold Signature*.
4.  **Rebind Sukses**: Kunci enkripsi baru diterbitkan pada perangkat keras baru penyintas, sedangkan hak akses pada perangkat lama otomatis dicabut melalui pembaruan status *Global Ledger state*.

## 5. Pelindung Privasi & Modul Hukum (Ethical Override Framework)

SSPA mengimplementasikan kontrol kedaulatan data di mana perlindungan privasi dapat diselaraskan dengan kebutuhan hukum internasional tanpa melanggar hak penyintas.

*   **Mekanisme Cryptographic Shredding**: Jika pemilik identitas memicu hak untuk dilupakan (*the right to be forgotten* / kepatuhan regulasi sejenis GDPR), sistem akan mengeksekusi penghancuran kunci enkripsi asimetris secara lokal di tingkat vault klien. Hasil data pada ledger global tetap ada demi integritas riwayat rantai blok, namun isinya selamanya menjadi *unreadable random noise* (tidak dapat didekripsi).
*   **Protokol Multi-Sig Dekripsi Pengadilan**: Data bukti pelanggaran kemanusiaan sensitif yang terenkripsi di *Off-Ledger Storage* hanya dapat dibuka untuk keperluan investigasi internasional pasca-krisis jika mendapatkan tanda tangan otorisasi gabungan dari pemilik data (konsensus penyintas) dan korps pengadilan hukum internasional terverifikasi.
